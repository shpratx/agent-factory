1. HLD vs LLD — Scope Distinction

HLD (High-Level Design)

Answers: "What components, how they interact, where they live"

Audience: Architects, tech leads, stakeholders

Granularity: System/service level

  • Solution overview & scope boundaries
  • Component architecture & responsibilities
  • Integration design (sync/async flows, protocols)
  • Data architecture (stores, ownership, flows)
  • Infrastructure topology
  • Security architecture & trust boundaries
  • NFR strategy (scalability, DR, performance)
  • Technology decisions with rationale
  • C4 Context & Container diagrams

LLD (Low-Level Design)

Answers: "How each component is built internally"

Audience: Developers, squad engineers

Granularity: Class/function/table level

  • Class/module design & patterns applied
  • API specification (endpoints, schemas, errors)
  • Database schema (tables, columns, indexes)
  • Sequence diagrams (method-level flows)
  • State machines & entity lifecycle
  • Algorithm detail & business logic
  • Error handling & retry configuration
  • Configuration & feature flags
  • C4 Component diagrams

The HLD agent's output is the LLD agent's most critical input — LLD implements what HLD prescribes.

Summary Comparison

Greenfield

  • Primary concern: Making the right choices from a blank slate
  • Critical KBs: L1 (standards + patterns + infra) + L2 (domain)
  • L3 dependency: Minimal — no prior state exists
  • Risk without KBs: Over-engineering or non-standard choices

Brownfield

  • Primary concern: Fitting new capability into existing constraints
  • Critical KBs: L3 (baseline + ADRs + contracts) + L2 (integrations)
  • L3 dependency: Heavy — prior state dictates every decision
  • Risk without KBs: Breaking existing systems or duplicating services

Detection Logic

Both agents should detect greenfield vs brownfield from input context:

  • Presence of kb-L3-{project}-application-baseline → Brownfield mode
  • Absence of L3 baseline KBs or explicit scenario: greenfield parameter → Greenfield mode
  • Brownfield: constrain to existing baseline first, then reference L1 patterns only where baseline doesn't prescribe
  • Greenfield: lean on L1 patterns to make fresh architectural/implementation decisions
