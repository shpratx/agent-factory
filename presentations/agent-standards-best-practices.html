1. The Two Agent Types: Core & Utility

Every agent in our ecosystem is classified as one of two types. This separation is the foundational architectural decision that enables reuse and composability.

C Core Agents

Do the actual work. Solve a specific, well-defined problem.

  • Platform-agnostic — zero bespoke logic
  • Unified input/output contracts
  • Pluggable into any workflow at any position
  • Reusable across clients and domains
  • Single responsibility principle

U Utility Agents

Wrap around Core agents. Handle cross-cutting and bespoke needs.

  • Platform-specific integrations (Jira, Confluence, Git)
  • Data transformation and formatting
  • Orchestration and routing logic
  • Logging, auditing, compliance wrappers
  • Client-specific customisations
Core agents are the brain — Utility agents are the hands that connect them to the world.

Why This Separation?

Core agents must be agnostic to any specific bespoke requirement or platform. They follow standard, consistent input and output contracts. The whole idea is that Core agents can be plugged in any sequence in any workflow — and that's fostered by unified contracts.

Utility agents wrap around Core agents and cater to bespoke requirements: "store the output to Jira", "publish to Confluence", "mask PII before processing", "hold for human approval". If the client changes from Jira to Azure DevOps, only the Utility agent changes — the Core agent remains untouched.

Anti-Pattern ✗

"The Vision Generator writes its output directly to a Confluence page and formats it in Atlassian markup."

This couples the agent to a platform. If the client uses Notion, the agent breaks. The generation logic is Core; the publishing is Utility. Split them.

Core or Utility? Decision Guide

It's a Core Agent if…

  • It solves a specific intellectual problem (generate, extract, analyse, validate)
  • Its logic is the same regardless of which tools the client uses
  • It would be useful to a different client without modification
  • It produces structured content as its primary output
  • Removing it from the workflow removes a capability

It's a Utility Agent if…

  • It moves, transforms, or stores data for a specific platform
  • It adds a cross-cutting concern (auth, logging, masking, gating)
  • Replacing it with a different implementation doesn't change the workflow's intellectual output
  • It needs API keys, OAuth tokens, or platform SDKs
  • Removing it from the workflow removes a connection, not a capability

Grey Area Resolution

If an agent does both (e.g., "generate stories AND push to Jira") — split it. The generation is Core, the push is Utility. One agent, one job.

3. Standard Output Contract

Every agent must produce an output JSON aligned with the standard schema. The content block carries the agent's output in two fields: items (structured data objects for downstream agents) and artifacts (documents, code, or assets for use across workflows). An agent may populate one or both depending on what it produces.

The AgentOutput Envelope

{
  "agent_id": "L1-inception-vision-generator",
  "execution_id": "exec-<uuid>",
  "workflow_execution_id": "wf-<uuid>",
  "status": "success | failed",
  "content": {
    "type": "<content_type>",
    "schema_version": "1.0",
    "items": { ... },
    "artifacts": [ ... ],
    "execution_summary": "• plain text bullets"
  }
}

Why This Matters

PrincipleWhat it enables
Predictable ShapeEvery downstream consumer knows exactly what to expect — same envelope, different content.type
Workflow Threadingworkflow_execution_id chains agents together. Each agent inherits it from upstream — enabling full traceability.
Self-Describingcontent.type tells the next agent what's inside without parsing the items. Route by type.
Status HandlingDownstream agents can branch on status without inspecting content — enabling graceful degradation.
Execution Auditexecution_summary captures what was done, which KBs were used, and what guardrails were evaluated — all in one place.

Items vs Artifacts

Items

Structured data objects consumed programmatically by downstream agents or systems.

  • Requirements, epics, stories, test cases
  • Manifest entries, entity lists
  • Analysis results, scores

Artifacts

Documents or assets used across workflows — often human-readable or stored externally.

  • Vision documents, design docs
  • Generated code files
  • Presentations, reports
One contract to rule them all. The output of any Core agent is valid input to any other.

4. README & spec.yaml for Every Artifact

Every artifact in the ecosystem — whether it's an agent, workflow, tool, guardrail, or knowledge base — must have two mandatory documentation files:

README.md

Depicts how the artifact is supposed to be used and what it composes of.

  • Purpose — why it exists (one paragraph)
  • What does it do? — detailed behaviour
  • How does it work? — processing pipeline
  • Input / Output — what goes in, what comes out
  • Composition — file tree showing structure
  • Dependencies — what it relies on

spec.yaml

Machine-readable specification of the artifact's contract, capabilities, and constraints.

  • Metadata — name, version, layer, phase, owner
  • Purpose — description, I/O, business value
  • Contract — input/output schemas, parameters
  • Context — knowledge bases, guardrails
  • Quality — evaluation ref, min scores
  • Versioning — schema versions, instruction hash

Applies to ALL Artifact Types

ArtifactREADME coversspec.yaml covers
AgentPurpose, I/O, composition, usage examplesContract, guardrails, KBs, quality thresholds. Also requires output_schema.json and evaluation.md
WorkflowPipeline description, agent sequence, triggersSteps, dependencies, timeout, retry policy
ToolWhat it does, invocation examples, auth requirementsAPI schema, rate limits, permissions
GuardrailWhat it checks, when it fires, severity levelsRules, thresholds, enforcement mode (block/warn)
Knowledge BaseDomain covered, sources, update frequency, quality barSchema version, content type, access level, layer
If it doesn't have a README and spec.yaml, it doesn't exist in the ecosystem. These are the minimum bar for discoverability and governance.

5. Reflection & Evaluation Criteria

Every agent must reflect upon its output before delivering it. This is not optional — it's a mandatory quality gate built into every agent's processing pipeline.

Reflection Process

  1. Generate — Produce initial output following processing rules
  2. Reflect — Check output against every item in the evaluation criteria
  3. Identify — Find gaps, errors, inconsistencies, or missed items
  4. Fix — Amend the output silently
  5. Deliver — Only return the final, corrected output

Interim reasoning and draft outputs are never exposed. Only the final result is delivered.

Evaluation Criteria: Two Rubrics

Every agent must have an evaluation.md covering both rubrics. This file is the single source of truth for what the agent checks during reflection — the reflection process (step 2: "Reflect") uses the checklist defined in evaluation.md as its verification criteria.

Capability Rubric

Does the agent do its job correctly?

  • Faithfulness — every output item traces to input (≥ 0.90)
  • Hallucination — no invented content (≤ 0.10)
  • Completeness — all expected items produced
  • Relevance — output is actionable (≥ 0.85)
  • Reasoning quality — decisions explained (≥ 0.80)
  • Citation completeness — every item cites source (≥ 0.95)

Agentic Rubric

Does the agent behave correctly as an agent?

  • Contract compliance — output matches schema exactly
  • Graceful failure — INSUFFICIENT_CONTEXT on bad input
  • Reflection evidence — summary shows what was found and fixed
  • KB reporting — all consulted KBs listed with content used
  • Guardrail compliance — all guardrails evaluated and reported
  • Idempotency — same input produces consistent output

What Reflection Must Report (in execution_summary)

  • What was produced (item counts, types)
  • Key decisions made
  • What reflection found and changed
  • Knowledge bases consulted — name and what content was retrieved
  • Guardrails evaluated — names and pass/fail
  • Tools invoked — names and outcome
  • Gaps or issues flagged
An agent that doesn't reflect is an agent that ships its first draft. We ship the final draft.

6. Examples & Golden Responses

Every agent must have examples and golden responses to compare and validate the output against. These serve as both documentation and automated validation benchmarks.

Examples (Minimum 2 pairs)

Input/output pairs demonstrating the agent's behaviour:

  • Happy path — realistic, detailed input producing full output
  • Edge case — empty, vague, or boundary input showing graceful handling

Each example must be valid JSON matching the contract schema and include the full AgentOutput envelope — not partial snippets.

Golden Responses (Minimum 2 pairs)

Production-grade benchmark responses used for automated evaluation:

  • Golden 01 — full success with realistic, detailed content
  • Golden 02 — failure/edge case showing correct error handling

Golden responses must be paired with corresponding input files and include evaluation criteria metadata.

File Structure

agents/{agent-name}/
├── examples/
│   ├── input-01-{scenario}.json       ← Happy path input
│   ├── output-01-{scenario}.json      ← Expected output
│   ├── input-02-{scenario}.json       ← Edge case input
│   └── output-02-{scenario}.json      ← Graceful failure output
└── golden/v1.0.0/
    ├── input-golden-01-{scenario}.json ← Detailed realistic input
    ├── golden-01-{scenario}.json       ← Production-grade benchmark
    ├── input-golden-02-{scenario}.json ← Failure scenario input
    └── golden-02-{scenario}.json       ← Correct failure handling

What Golden Responses Must Demonstrate

RequirementPurpose
Full AgentOutput structureValidates schema compliance end-to-end
Metadata on every item (confidence, reasoning, citation)Proves grounding and traceability
execution_summary with reflection findingsProves the agent reflected and reported
INSUFFICIENT_CONTEXT handlingProves graceful failure without hallucination
Validates against output_schema.json without errorsProves contract integrity

INSUFFICIENT_CONTEXT — The Failure Pattern

When input is empty, gibberish, or too vague to process, agents must not hallucinate. They return a structured failure:

{
  "agent_id": "L1-inception-vision-generator",
  "execution_id": "exec-<uuid>",
  "workflow_execution_id": "wf-<uuid>",
  "status": "failed",
  "content": {
    "type": "product_vision",
    "schema_version": "1.0",
    "items": {},
    "artifacts": [],
    "execution_summary": "• Produced 0 items — INSUFFICIENT_CONTEXT.\n• Input too vague to extract actionable content.\n• Suggested: ask stakeholder for [specific missing detail]."
  }
}

Versioning Golden Files

Change TypeVersion BumpAction
Prompt wording tweak (same behaviour)Patch (1.0.0 → 1.0.1)Update instruction_hash
New output field added (backward compatible)Minor (1.0.0 → 1.1.0)New golden set in golden/v1.1.0/
Output structure change (breaking)Major (1.0.0 → 2.0.0)New golden set + update output_schema.json
Golden responses are the contract. If an agent's output doesn't match its golden standard in structure and quality, it doesn't ship.

7. Naming Convention

All artifacts in the ecosystem follow a consistent naming pattern that encodes their layer, domain, and function — enabling discoverability and governance at scale.

Agents

L{layer}-{phase}-{function}

L1 = Enterprise (generic, any domain)       e.g. L1-inception-vision-generator
L2 = Domain/LOB (domain-specific)           e.g. L2-payments-inception-story-generator
L3 = Project (project-specific)             e.g. L3-kyc-inception-requirements-extractor
L4 = Squad (team-specific)                  e.g. L4-alpha-construction-code-reviewer

Knowledge Bases

kb-L{layer}-{domain-or-topic}

e.g. kb-L0-enterprise-architecture
     kb-L1-springboot-standards
     kb-L2-payments-domain
     kb-L3-kyc-project-context

Guardrails

gr-L{layer}-{check-type}

e.g. gr-L1-output-schema-validator
     gr-L1-pii-detection
     gr-L2-payments-compliance
     gr-L3-hallucination-detector

Workflows

wf-L{layer}-{phase}-{pipeline-name}

e.g. wf-L1-inception-idea-to-backlog
     wf-L2-payments-design-to-lld
     wf-L3-kyc-full-sdlc

Tools

tool-{platform}-{action}

e.g. tool-jira-create-ticket
     tool-confluence-publish-page
     tool-github-create-pr
     tool-slack-notify-channel

Core Artifacts: L1 and L2

  • L1 — reusable across all clients and domains
  • L2 — reusable within a domain (payments, lending, retail)
  • Never client-specific

Utility/Bespoke Artifacts: L3 and L4

  • L3 — project-specific (client integrations, custom workflows)
  • L4 — squad/team-specific (local overrides, experiments)
  • Tools are always platform-prefixed, not layered

8. Knowledge Bases & Runtime Context

Core agents don't hardcode domain knowledge — they declare which KBs they need in spec.yaml and the orchestrator injects them at execution time.

PrincipleDetail
Injected at RuntimeThe agent never fetches KBs itself. They are attached by the orchestrator.
Layered KnowledgeL0 = enterprise architecture, L1 = best practices, L2 = domain-specific, L3 = project context
Reported in SummaryEvery agent must list which KBs it consulted and what it retrieved from each
Quality DependencyAgent output is only as good as its KBs. Golden KBs are curated, version-controlled, and validated.
Agents don't know things — they consult knowledge bases. Change the KB, change the output. No code changes needed.

9. Pre-Publish Checklists

Every artifact must pass its corresponding checklist before it is published to the ecosystem. No exceptions.

Agent Checklist

  • ☐ Name follows L{N}-{scope}-{phase}-{function}
  • ☐ Layer + ownership identified
  • ☐ If L2/L3/L4: extends parent (not copy-paste)
  • ☐ KBs appropriate for layer
  • ☐ Guardrails attached
  • ☐ Output schema defined (output_schema.json)
  • ☐ Uses standard AgentOutput envelope
  • ☐ Zero platform-specific logic (Core agents)
  • ☐ Includes execution_summary with KB/guardrail/tool reporting
  • ☐ Self-reflection before output delivery
  • ☐ INSUFFICIENT_CONTEXT handling for bad input
  • ☐ Has spec.yaml, evaluation.md, output_schema.json, README.md
  • ☐ Examples (minimum 2 input/output pairs)
  • ☐ Golden responses (minimum 2 benchmark pairs)
  • ☐ Golden files validate against schema
  • ☐ Tested with representative inputs
  • ☐ Documented

Workflow Checklist

  • ☐ Name follows L{N}-{scope}-WF-{phase}-{outcome}
  • ☐ Agents at correct layer
  • ☐ Tools at L1 (unless domain-specific)
  • ☐ Quality gates between handoffs
  • ☐ HITL (Human-in-the-Loop) points defined
  • ☐ Error handling per step (retry, fallback, or escalate)
  • ☐ Idempotent (safe to re-run)
  • ☐ Versioned (semver)
  • ☐ Has README.md and spec.yaml
  • ☐ Documented (pipeline description, agent sequence, triggers)

Tool Checklist

  • ☐ Name follows tool-{platform}-{action}
  • ☐ L1 unless domain-specific system
  • ☐ Credentials via SecretsSDK (zero hardcoded)
  • ☐ Input validation before API call
  • ☐ Structured ToolResult (success, data, error)
  • ☐ Timeout configured (30s default)
  • ☐ Idempotent creates (check-before-create)
  • ☐ Tested with mock + real API
  • ☐ Has README.md and spec.yaml
  • ☐ Documented (invocation examples, auth requirements)

Contract Checklist

  • ☐ Input contract defined (accepted sources, required/optional params, file formats)
  • ☐ Output follows AgentOutput schema (agent_id, status, content.type, items, artifacts)
  • ☐ Output schema versioned (schema_version field)
  • ☐ Backward compatible with prior version (no removed/renamed fields)
  • ☐ workflow_execution_id populated for traceability
  • ☐ Failed status returns empty items (no hallucinated partial data)
  • ☐ Schema validated at runtime (input + output)
  • ☐ Tested with upstream agent output as input (workflow composition verified)

Knowledge Base Checklist

  • ☐ Name follows kb-L{N}-{domain}-{scope}
  • ☐ Layer + ownership identified
  • ☐ Source documents identified and access approved
  • ☐ Data classification reviewed — no PII or restricted content without approval
  • ☐ Chunking strategy defined (size, overlap, semantic boundaries)
  • ☐ Embedding model selected and configured
  • ☐ Retrieval tested — precision and recall meet threshold (≥ 85%)
  • ☐ Grounding rules defined (which agents access this KB)
  • ☐ Freshness schedule established (re-ingestion cadence)
  • ☐ Versioned (changes tracked, rollback possible)
  • ☐ Visibility rule respected (L3 sees L1+L2+L3; L2 sees L1+L2 only)
  • ☐ Has README.md and spec.yaml
  • ☐ Documented (sources, update cadence, owner, consumers)

Guardrail Checklist

  • ☐ Name follows gr-L{N}-{check-type}
  • ☐ Layer + ownership identified
  • ☐ Rules and thresholds defined
  • ☐ Enforcement mode specified (block or warn)
  • ☐ Severity levels documented
  • ☐ Tested with passing and failing inputs
  • ☐ Has README.md and spec.yaml
  • ☐ Documented (what it checks, when it fires)

10. Token Optimisation Techniques

Agents operate within LLM context windows. Unoptimised agents consume 35,000–40,000+ tokens per execution (instructions + KBs + output), risking context overflow and degraded performance. These techniques reduce token consumption by 60–77% without compromising output quality.

Technique Summary

IDTechniqueToken SavingsQuality Impact
S1Chunk and Retrieve (RAG)~80% of KBNone — retrieves only relevant chunks
S2Condense Evaluation KB~70% of evalNone — checklist format preserves all criteria
S3Slim Domain KB~70% of KBNone — retains PM implications, journeys, state machines, glossary
S4Template as Instruction~100% of template KBNone — structure embedded directly in prompt
S5Two-Phase GenerationImproved attention allocationPositive — prevents "lost in the middle" problem
S6Evaluation at Validation~100% of eval during generationNone — evaluation happens post-generation via separate validator

S1: Chunk and Retrieve (RAG)

Don't pass the full domain KB. Use embeddings to retrieve only relevant chunks based on the user's input.

  • Before: Full KB (~25,000 tokens) passed every execution
  • After: Top-10 relevant chunks (~3,000–5,000 tokens)
  • Requires: Vector store, embedding model, retrieval pipeline
  • When: Domain KB > 5,000 tokens

S2: Condense Evaluation KB

Evaluation docs are often verbose with rubric tables and lengthy explanations. Compress to checklist-only format.

  • Before: ~7,000 tokens (verbose rubrics, explanations, examples)
  • After: ~500 tokens (quality gates + scores table + reflection checklist)
  • Remove: Score range descriptions, redundant writing rules (already in prompt Don'ts), verbose reflection process steps
  • Preserve: All gate criteria, all threshold scores, all checklist items

S3: Slim Domain KB (Vision-Optimised)

Create layer-appropriate KB variants. Vision/requirements agents don't need field-level schemas — that's for LLD/code agents.

  • Before: ~24,000 tokens (full domain KB with data models, validation rules, API details)
  • After: ~7,400 tokens (PM implications, user journeys, state machines, glossary)
  • Keep (for vision/inception): PM implications (what to build), user journeys (full), state machines (full), glossary (full), section headings
  • Remove: Field-by-field data models, validation regex, implementation guidance, regulatory full-text
  • Naming: kb-L2-{domain}-slim.md alongside the full kb-L2-{domain}.md

S4: Template as Instruction

If a template KB is ≤ 3,000 tokens, embed its structure directly into the prompt. Eliminates a separate KB load.

  • Before: Template as separate KB (~1,200 tokens) + "fill template" instruction
  • After: Structure headings with inline requirements embedded in prompt (+150 tokens to prompt, -1,200 tokens net)
  • Format: List each section heading with its requirements inline (e.g., ### Feature Areas — ≥4 areas, each: description, capabilities, user value)
  • When: Template KB is structural (headings + placeholder guidance), not content-heavy

S5: Two-Phase Generation

Split processing into Phase 1 (extract relevant facts from KB) and Phase 2 (generate output from condensed brief). Both happen in a single model call.

  • Phase 1: Scan KB → extract only relevant user roles, business rules, terminology, systems, risks → produce internal brief
  • Phase 2: Generate full output from brief + document structure
  • Why: Focuses model attention — prevents "lost in the middle" where mid-KB content gets ignored
  • Shortcut: If KB < 2,000 words, skip Phase 1 (KB is already concise enough)
  • Rule: Phase 1 brief is internal only — never shown to user

S6: Evaluation at Validation (not Generation)

Don't pass the evaluation KB during generation. Use a separate validator agent post-generation to check quality.

  • Before: Evaluation KB (~7,000 tokens) loaded during generation
  • After: 0 tokens during generation; evaluation KB loaded only by validator agent
  • Architecture: Core Agent (generates) → Validator Agent (evaluates against rubric)
  • When: Context window is tight and evaluation criteria are complex

Recommended Combinations

Minimum Optimisation (S2 + S4)

Quick wins, no architecture changes.

  • Condense evaluation to checklist
  • Embed small templates in prompt
  • Savings: ~30-40%

Full Optimisation (S2 + S3 + S4 + S5)

Maximum reduction, single-agent architecture.

  • All quick wins + slim KB + two-phase
  • Savings: ~70-77%
  • Example: 37K → 9.6K tokens

When NOT to Optimise

  • KB is already < 3,000 tokens — overhead of slimming exceeds benefit
  • Agent produces code (needs full field-level schemas from KB)
  • Context window is 128K+ and execution is well within budget
  • Agent is in development/testing phase — optimise after quality is proven

Measuring Current Token Consumption

Before optimising, measure your agent's current context usage:

  • Rule of thumb: 1 token ≈ 4 characters of English text
  • Measure: sum character counts of all KBs + instructions + expected output, divide by 4
  • Target: total context (input + output) should be < 50% of model's context window
  • Red flag: if total exceeds 70% of context window, optimisation is mandatory
  • Quick wins: remove UUID format examples from prompts (models know UUID); remove verbose reflection step descriptions; collapse rules that repeat Don'ts
